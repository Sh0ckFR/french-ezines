Je voudrais apporter une suite à cet article en parlant 
rapidement des limites du hacking.

Le hacking est ( pour moi en tout cas ) une sorte de sport ou de jeu.
Mais si on commence à faire du carding et à arnaquer  des gens qui 
t'ont rien fait il n'y a pas de quoi se sentir fier.
Par contre , si on se sert d'un N° de CB, on envoie une lettre à la victime
et à la banque disant qu'on a hacké un compte ( et on donne le n° et tout et tout ).
Alors là la banque est ( théoriquement ) obligée de rembourser le pauvre pigeon
ce qui ne me fait pas pleurer car alors que ces gros PDG jouent à faire des OPE de
plusieurs milliards, des gens ne mangent pas toujours à leur faim.
Mais arnaquer quelqu'un dont tu ne connais pas les ressources financières 
n'est rien que de l'ESCROQUERIE.

C'est pourquoi il faut se servir de notre esprit rebelle ( :o) ) pour arnaquer
ceux qui détiennent l'argent et le pouvoir pas les pauvres travailleurs qui t'ont rien fait.
C'est donc là que je suis d'accord avec Crash :
Arnaquer FT, les banques, ça ne leur fera pas de mal.

Pour ce qui est de la question de l'éthique du hacker, c'est vrai qu'on devrait se mobiliser
avec tous ceux du groupe pour établir une liste noire des sites pour ensuite les hacker.
Le message est donc lancé et ceux qui veulent y participer peuvent nous contacter.
Il ne tient qu'à nous d'améliorer notre image de marque !

Slugishness, hang-ups, freezing, unexpected crashes, erratic mouse movement. 
All these can result from the use of Windows 98's generic VMM32.VxD driver instead of the 
device specific VxD drivers that come with Windows 98. 

Think of VMM32.Vxd as a driver that knows how to operate 'in a general sort of way' with 
any device. A specific VxD driver knows 'exactly' how to operate with the device it was 
designed for. 

Windows 98 does not install all the device specific drivers and may, incorrectly, rely on 
the non-specific VMM32.VxD driver to manage different devices. An early report suggests 
that the newer Windows 98SE is also at fault. 

A Virtual Device Driver (VxD) is a special type of Windows driver that allows Windows to 
perform functions at optimal CPU speed, and permits interaction with the hardware and internal 
Windows functions. 32 bit Virtual memory manager device(VMM32.VxD) is the VxD file that 
provides the primary functions (kernel) in Windows. VxDs sometimes conflict and can lock 
up the system. 

Your system will be faster, and safer, by using the device specific VxDs. Specific drivers are 
always safer, and optimized for better performance. 

Checking your current drivers:

2. Highlight System board and click Properties, Driver tab, Driver File Details 
31 If you see C:\Windows\System\configmg.vxd you are using the correct VxD driver

4. Now check (as above) your modem, mouse, etc

vcomm.vxd communications driver

vflatd.vxd video flat-frame buffer virtual device 
vdmad.vxd DMA controller driver 
vmouse.vxd mouse driver 
configmg.vxd configuration manager for the motherboard 
ntkern.vxd kernel driver used for the universal host controller 
You may not need all of these.

I)NUMEROTATION

Il existe 2 types ( ou peut être plus mais je les connais pas ) de numérotation :

a) "Vocale"
b) "Par impulsions"

Avant d'expliquer allez faire un tour dans :
Panneau de Configuration/Modems/Propriétés de numérotation
et regardez " Composer en utilisant la numérotation " et là vous avez le choix :

	O Vocale       O Par impulsions

Vous devez avoir normalement la configuration suivante :

	0 Vocale       O Par impulsions


a) Vocale

Ce type de numérotation envoie des fréquences directement dans la ligne,
c'est commme une sorte de mélodie si vous préférez.
On appelle ces fréquences les fréquences "DTMF".
C'est une numérotation rapide. On utilise celle-ci sur la plupart des téléphones
actuels. ( + de 2 ou 3 ans )

Exemples : - PHILLIPS SKIP 300 ( téléphone "fixe" )
           - Amarys de F.T ( " sans fil " tiens tiens je peux pas m'empêcher 
             de parler de FT dans mes articles c'est plus fort que moi :o(   )

b) Par impulsions

Ce type de numérotation envoie des impulsions dans la ligne, qui ressemblent à
une sorte de grattements rapprochés ( TAC-TAc-TAC-TAC..........TAC-TAC-TAC-TAC).
Plus le chiffre croît, plus le temps de numérotation est long. 
Le 0 (zéro) est le chiffre le plus "long", le 1 le chiffre le plus "court".
Donc, pour ceux qui n'auraient pas encore compris la classification par longueurs
croissantes est :
1,2,3,4,5,6,7,8,9,0
Certains anciens téléphones utilisent encore la numérotation par impulsions.
 
Exemples : - MATRA TM1 ( téléphone "fixe" )
           - MATRA NAUTILA 100 ( "sans fil" )


Chez certains téléphones ces impulsions sont plus ou moins audibles.
Des fois vous n'entendez que le début et la fin de chaque impulsion, par
exemple :
  
TAC= audible
  (TAC)= quasiment inaudible


TAC (TAC) (TAC) ....... (TAC) (TAC) TAC


D'autres fois encore on entend que le début :


TAC (TAC) (TAC) ....... (TAC) (TAC) 


Et encore d'autres fois ( et c'est le cas le plus courant ) on entend
toutes les impulsions : 
TAC TAC TAC ....... TAC TAC TAC


Si votre modem possède un haut-parleur faîtes le test en changeant les 
propriétés de numérotation vous allez tout de suite saisir la différence.
A noter que les téléphones à impulsions sont à déconseiller pour le phreak 
sur VMB/PABX car même si le numéro est composé normalement la VMB ne reconnaît pas les 
commandes que vous tapez et ne réagit pas.
Par exemple vous faîtes le 0800 XX XX XX ( non je donne pas de numéros

	Faites une copie de 'NetTerm.exe' (ex: 'NetTerm.org.exe') et désassembler la,

 Tout porte à croire que pour nous enregistrer il faudra executer la seconde fonction. 
 Parfois on ne trouve pas de fonctions à côtés pour nous aider, mais nous avons souvent des 
 String Références qui (comme ici) sont directement liés à l'enregistrement (ex: 'Legal', 
 'OWNER', 'Unregistered Copy', 'CODE'). On a même une chaîne qui pourrait être un serial number
 (dans ces cas là, il faut mieux le tester, parfois c'est un code unique et vous aurez même 
 pas besoin de cracker le produit).. Mais ce n'en est malheuresement pas un. 
 	Bref.. on s'éloigne un peu là.. reprenons donc, on est certainement au bon endroit. 
 Le but est de trouver si possible le cas de figure typik d'un saut résultant d'une routine de
 vérification indiquant si le serial est bon ou non. En gros, c'est pour savoir à quelle 
 adresse le prog doit aller par rapport au infos saisies. Généralement en asm, ceci se traduit 
 assez souvent par ceci : 

  call <adr>                           ;  Execute une Sous-Routine à l'adresse <adr>
                                       ;     --> Test si notre code est bon ou pas
  mov <op1>, <op2>                     ;  Met dans <op1> la valeur de <op2>
                                       ;     --> Garde en mémoire la réponse de ce test
  cmp <op1>, <op3>                     ;  Compare <op1> et <op3>
  		                           ;     --> Test si la réponse est bonne ou pas 
  je <adr>                             ;  Saute à l'adresse <adr> si la comparaison est égale
                                       ;     --> Se place dans la partie Bon ou Mauvais Code
  
Remarque : Ce schéma type n'est pas unique (heureusement sinon on s'ennuierait rapidement). 
°°°°°°°°   De nombreuses variantes existent, parfois totalement différentes. Entre autres au 
           niveau du dernier saut, cela varie beaucoup, il peux y avoir plusieurs conditions
           de saut par exemples :
                   je   = Jump if Equal       -->  Saut si égal
                   jne  = Jump if No Equal    -->  Saut si différent
                   jz   = Jump if Zero        -->  Saut si egal Zéro
                   jnz  = Jump if No Zero     -->  Sauf si different de Zéro

>>> Analyse de la Routine <<<
<<<<<<<<<<<<<<<<<<<<<<<<<<<<<
  
Etudions le code autour de Notre String Ref 'Register' :

 	Ok, alors là c'est presque parfait, on a bien le schéma typique mais on en a deux.
 Pourquoi deux routines de tests l'une après l'autre ? Faut-il modifier les deux ?
 Et bien, quand on sait pas, le mieux c'est de tester, mais on peut également réfléchir
 quelques secondes... j'vous laisse réfléchir... encore un peu... bon... vous n'avez
 pas trouvé ? C'est pas grave, on remarque que ce n'est pas deux routines identiques,
 les calls sont différents, le 1er correspond sûrement à une routine de test pour savoir
 si nous sommes déjà enregistrés. Si c'est le cas, la routine de test qui suit n'est pas
 à executer, on peut sauter directement en 4303C5 qui est la fin de la routine globale.
 Par défaut, le prog n'est pas enregistré, il faut passer par ce qui suit, donc, dans
 ce 1er morceau on ne change rien. Par contre, le second schéma typik, est beaucoups plus 
 intéressant, de part son nom, on comprendra bien de kwa il s'agit..
 
 Alors, étudions cela de plus près...
 
1) Le 'call' va nous dire si le code qu'on a rentré est correct ou non, pour ça il va devoir
   changer l'état d'un registre (Note: un registre c'est un peu comme un emplacement mémoire
   temporaire, il permet de stoker une valeur), ici on comprend facilement que c'est le registre 
   'eax' (Note: il y a  beaucoup d'autres registres ex: ebx,ecx,edx...), car on l'utilise à la 
   ligne qui suit.
 
2) Le 'mov' va garder en mémoire (certainement pour l'utiliser plus tard) ce qu'il y a dans 
  'eax', la réponse de notre 'call' (pour savoir si le code est ok) qui est généralement
   représenté numériquement par 0 (=mauvais) ou 1 (=bon).
 
3) Le 'cmp' va donc comparer cet emplacement mémoire à la valeur 0, pour savoir si l'exécution
   du programme doit aller vers la routine correspondant au bon Code ou à celle du mauvais.
 
4) Le 'je' fait un saut si la réponse est égale à 0, va à l'adresse 4303B4 si la réponse est 
   égale à 0, donc logiquement si la réponse est mauvaise. A comprendre par là que notre call 
   à indiqué que le mot de passe est faux, et qu'il faut aller dans la partie code erroné.
 
 	Bah voilà.. là on a touché le point faible du soft... :)) Enjoy YourSelf.. =))
 Maintenant, il nous suffit de modifier cette partie à notre guise pour qu'il ne saute pas à 
 l'adresse 4303B4 (routine mauvais code), mais qui continue dans la routine bon code. Alors 
 pour ça, il y a plusieurs méthodes, on peut par exemple remplacer la commande 'je' par des 
 'nop' (=No OPeration) qui ne font rien, NetTerm suivra donc le chemin du bon code. Même si 
 c'est faisable, mais ce n'est pas très propre car la commande 'je' est codée sur 6 octets 
 (0F-84-61-00-00-00), et il nous faudrait donc écrire 6 'nop' car cette commande est codé sur
 un seul octet (90). De plus, certains programme vérifient s'il n'y as pas trop de NOP à la 
 suite. Si c'est le cas, ils trouvent cela pour le moins byzarre et vous demande d'arrêter de 
 pirater leur pauvre petit programme. Sinon, le plus rapide est d'inverser le saut : au lieu 
 de sauter à l'adresse Mauvais Code si le code est faux, on veut le faire sauter à l'adrese 
 Mauvais code si le code est bon ! Pour ça, il suffit de remplacer 'je' (=saut si egal) par 
 'jne' (=saut si different)... Dans ce cas, le programme sautera à la routine Mauvais Code 
 seulement si le code est bon !! Ainsi, tout les mauvais codes fonctionneront :)))
  


>>>>>>>>>>>>>>>>>>>>>>>>>>>>>
>>> Modifications du Prog <<<
<<<<<<<<<<<<<<<<<<<<<<<<<<<<<

        Voilà pour la théorie, passons maintenant à la pratique, comment faire pour changer 
 une instruction ? Bon, il faut d'abord savoir comment on la change, on a donc décidé de 
 mettre 'jne' à la place de 'je'. Ceci est propre et pratique, on ne change qu'un seul octet, 
 à la place de 0F-84-61-00-00-00 on va mettre 0F-85-61-00-00-00. Vous vous demandez sûrrement 
 d'où je sors mon 85 ? Bah.. pour ça il faut connaitre les OpCodes (=Codage) des commandes Asm, 
 mais vous verez c'est souvent les même jumps qui reviennent. Voici un petit tableau d'opcodes 
 pour les jump les plus communs selon si le saut est short ou near :

 -------------------------------------------------                        
|      Type de Jump       |   Short   |   Near    |    
|-------------------------+-----------+-----------|
|                         |           |           |
| JE   (Jump if Equal)    |     74    |  OF-84    |
| JNE  (Jump if No Equal) |     75    |  0F-85    |
| JZ   (Jump if Zero)     |     74    |  0F-84    |
| JNZ  (Jump if No Zero)  |     75    |  0F-85    |
|                         |           |           |
 -------------------------------------------------
 
	Pour nous, on est dans le cas d'un Jump Near, il faut donc remplacer 0F-84 (JE) par
 0F-85 (JNE). Lancé HexWorks ou un autre éditeur Hexadécimal pour modifier directement les
 valeurs hexa du fichier. Pour se placer au bon endroit, vous pouvez soit noter une longue 
 suite d'opcodes des parages pour y faire une recherche et arriver dessus, ou alors, et c'est 
 le mieux à faire (merci White..), récupérer l'offset de la commande, pour ça, avec W32Dasm, 
 doublez cliquer sur la ligne où se trouve notre 'je' (43034D), et noter ce qu'il y a d'inscrit
 dans la barre d'état : '@Offset 0002F74Dh'. Allez dans le menu Edit/Goto (Alt+F5) de HexWorks, 
 sélectionner les options 'Hex' et 'Beginning of file', entrez notre offset '2F74D', et Hop Là !
 On arrive pile poils dessus ! Maintenant, on a sur le curseur, la suite d'octets (réprésentant
 l'opcode) de notre commande 'je 004303B4'. On modifie maintenant l'instruction, je répète donc, 
 de 0F-84 (je) en 0F-85 (jne). Vous fermer ensuite votre éditeur héxa. en prenant soins de bien
 sauver les changements.

 On a le message 'Key Verified' dans la barre des taches, on va dans le menu About, et on voit

	Ceci est généralement dû a un test au lancement du soft pour vérifier si votre nick 
 fonctionne avec le code entré (qui ont été enregistrés dans le fichier 'NetTerm.ini'). Dans 
 ce cas, le programmeur a sûrement fait appel à la même routine de test, donc au même 'call'. 
 Il nous faut retrouver dans le programme un autre appel à ce 'Call 0044AEEA', on peut très 
 bien faire une recherche avec l'option 'Find Text' du menu 'Search' de W32Dasm, mais je vais 
 vous expliquer une autre méthode bien plus pratique. Placer vous sur notre call, et cliquer
 sur le bouton 'Call' qui permet d'aller directement aux instructions de ce dernier, un peu 
 comme si on rentrait à l'interrieur de la commande. On s'apperçoit qu'il n'est composé que
 d'un saut vers une adresse inaccessible. En regardant juste au-dessus de notre curseur, on 
 peut lire ceci :

:00430199   , :0043033B   

	Ce qui veut dire que notre Call se fait appeler par deux adresses, le notre (43033B) 
 et un autre (430199) qui est celui à également modifié. On va donc à cette adresse (menu Goto,
 Goto Code Location), et l'on voit sous nos yeux une nouvelle structure typik :

>>> Execution du Prog Re-Modifié <<<
<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<<

	Ok, maintenant on relance le soft, on peut tout tester, la routine de Code écrit votre
 nick avec votre code dans le fichier 'NetTerm.ini'. Il fait une vérif au départ, mais comme 
 l'on vient de cracker cette partie, le soft est directement cracké de part notre nick/code 
 déjà inscrit dans le .ini, mais sinon, le soft est correctement cracké. Be Happy !! =)))

	Une Remarque importante, on aurait pu directement modifier la partie à l'interrieur du
 call pour ne faire qu'une modification ciblée à un endroit. Pour ici, nous avions que deux
 endroit à modifier, mais s'il y avait plusieurs appels à ce Call, il aurait fallut modifié
 à chaques appels les structures typik. Donc, le mieux est de toujours modifier le call même,
 nous ne l'avons pas fait ici, d'une part beKoZe on n'avait que deux appels, et d'autre parts,
 car dans le call, on jump directement sur une adresse mémoire inconnue ! Mais en fait, pas si
 inconnue que ça, en fait on donne la main à la fonction ISILOGO.DoRegistration. Donc, si l'on 
 souhaite mieux cracker le prog, il faut désassembler le fichier ISILogo.dll. Une fois arriver 
 dedans, vous cliquer sur le bouton 'Exp Fn' (= Exported Fonctions) pour aller sur notre notre
 routine 'DoRegistration', ensuite il faudrait étudier la routine, pour qu'au lieu de terminer 
 avec eax = 0 pour un mauvais code, qu'il soit toujours différent de 0, en le mettant à 1 par
 exemple. Vous pouvez essayer vous même, mais nous stoppons notre cours ici, pour ne pas rentrer 
 dans les détails techniques sur l'assembleur.

 que chacuns fait comme bon lui semble, de part ses habitudes et préférences. Pour nos prochains

 pour mieux comprendre ce que vous lirez.. Ahh oui, surtout, après un crack réussi, vérifier à

 net (gratuitement bien sûr..). Il y a pas mal de p'tit programmes comme DCE (Data Crack Engine)
 qui vous permettent de faire les différences entre votre fichier original et celui cracké, pour
 créer un executable tout petit que vous pourrez alors donner très facilement, ce dernier se 
 chargera d'appliquer vos changement sur le fichier original.
